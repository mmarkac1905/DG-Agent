## Pipeline mental map (for orientation in the new session)

| Stage | What happens | What the LLM sees | Output |
|---|---|---|---|
| **0. Term spec** | User writes name, definition, grain, filter, unit | (no LLM — user input) | `business_glossary` row, status=`draft` |
| **A. Scope derivation** | LLM picks source SAP tables | `sap_data_dictionary` (42 tables × 326 cols), `org_structure`, `domain_facts` | `s2t_mapping.source_table` rows, status → `scope_confirmed` |
| **B. Blocker analysis** | System + LLM find gaps (unknown joins, missing cols) | Stage A scope + `sap_data_dictionary` + ABAP catalog | Blockers with addressed/open state |
| **C. Domain EDA** | Real-data queries: distinct values, nullability, joinability, distributions | None — pure DuckDB | `domain_facts` (reusable) + `analysis_findings` (per-term) |
| **C'. Term EDA** | Term-specific checks: does the filter return rows? does the grain hold? | (glue checks before D) | `analysis_findings` |
| **D. S2T generation** | LLM writes full mapping rows + dbt SQL | A + B + C + C' + `abap_logic_catalog` + `dbt_column_lineage` + `archive_log` (prior attempts) | `s2t_mapping` rows + `dbt/models/.../<term>.sql` |
| **E. Deploy** | dbt seed → run → test → semantic validator (grain/filter/unit gate) | (validator, not LLM) | Mart materialised, status → `approved` |

The stage the user most wants to understand: **C (Domain EDA)** — what queries run, how their results shape the Stage D prompt.

## Reasoning chain that led here

1. User wanted to learn the system end-to-end via a real term build.
2. Proposed candidate: combine finance (BSEG/BKPF) with operational data for a profitability-like metric. User open to extending sample data.
3. Inventory check revealed: BSEG/BKPF loaded but only at staging; no vault/marts; only GR-side postings present.
4. SD module absent entirely → true profitability not possible without data extension.
5. User pushed back on the proposed KPI ("cost per deployed device"): how is it different from existing total procurement spend? — **It isn't a different data source, it's a derivative ratio (normalised for volume). Teaches the pipeline but is not strongly differentiated as a business KPI.**
6. User asked: aren't invoices in FI, not MM? — **Correct. SAP invoices have dual representation: MM-IV (RBKP/RSEG) holds the verification record with PO reference; FI (BKPF/BSEG with BLART='RE') holds the journal entry / vendor-payable side. Our synthetic data only has the MM-IV side and the GR-side FI postings — the invoice-side FI postings are missing.**

## Pending decision (where the next session picks up)

Pick one of these three paths and the walkthrough begins from Stage 0:

### Path 1 — Stay in MM/MM-IV
- Build a term using existing money data already in marts
- No data-gen extension, no FI vault work
- Examples: cost of returns, stock-aging cost, price-per-unit variance vs benchmark, three-way-match break rate in €
- **Teaches**: full Stage A–E pipeline cleanly, no infrastructure expansion
- **Effort**: smallest

### Path 2 — Extend FI for completeness
- Extend the sample-data generator to produce invoice-side FI postings (BLART='RE')
- Build the finance vault layer (`hub_accounting_doc`, `link_invoice_accounting`, `sat_accounting_header`, `sat_accounting_line`)
- Add a governed reference seed: `gl_account_master` (account → category: cost / revenue / asset / liability)
- Build a finance fact (e.g. `fact_gl_cost_postings`) joining to operational dimensions
- **Teaches**: vault extension from staging, governed reference seed lifecycle, finance ↔ operational join in a mart, full Stage A–E
- **Effort**: medium — multi-session

### Path 3 — Add SD entirely for real profitability
- Path 2 + sample-generate VBAK/VBAP/VBRK/VBRP/KNA1
- Plus their full pipeline through OBT
- Enables real revenue + cost = profitability per material / customer / service
- **Teaches**: everything above plus new-module integration end-to-end
- **Effort**: largest — multi-session, biggest data-gen extension

**Recommendation**: Path 1 if the goal is fastest understanding of the LLM pipeline. Path 2 if the goal is also understanding vault/seed governance/finance integration. Path 3 if true profitability analytics is the eventual product target.
